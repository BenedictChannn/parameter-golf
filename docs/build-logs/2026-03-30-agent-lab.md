## Tranche H Design

- The next tranche is [`T-20260330-H`](../../agent_lab/tranches.md).
- Goal:
- ask whether the residual-control family is now overbuilt on the current best line.
- Why this family now:
- output-path local tuning flattened
- residual controls and skip topology are still under-tested
- they are distinctive to this script
- and they may be simplifying targets that save bytes, complexity, or optimization noise

- I added trainer env knobs so these can be tested without rewriting the model each time:
- `USE_RESID_MIX`
- `USE_ATTN_SCALE`
- `USE_MLP_SCALE`
- `SKIP_MODE=learned|unit|off`

- Planned residual-control experiments:
- `H1`: remove `resid_mix`
- `H2`: remove learned residual scales
- `H3`: keep skips but replace learned skip weights with unit skips
- `H4`: remove skip topology entirely
- `H5`: test a coherent simplification package

## Tranche H Progress

- `AL-20260330-006` tested `H1`: `USE_RESID_MIX=0`
- `final_int8_ttt_lora val_bpb: 1.3763`
- `1697` training steps in `600075ms`
- `Total submission size int8+zlib: 15816966 bytes`
- conclusion:
- this was not a near miss
- removing `resid_mix` regressed badly
- learned input-stream mixing is still materially helping on the current best line
- immediate implication:
- keep `resid_mix` on
- move to `H2` next instead of spending more compute on the same sub-question

- `AL-20260330-007` tested `H2`: `USE_ATTN_SCALE=0, USE_MLP_SCALE=0`
- `final_int8_ttt_lora val_bpb: 1.3666`
- `1685` training steps in `600007ms`
- `Total submission size int8+zlib: 15639317 bytes`
- conclusion:
- this also lost clearly, but less badly than `H1`
- learned residual scales still appear to help on the current frontier
- immediate implication:
- do not pivot back into “plain unit residuals” as a likely win
- `H3` is still worth running because skip-path parameterization is a different question

- `AL-20260330-008` tested `H3`: `SKIP_MODE=unit`
- `final_int8_ttt_lora val_bpb: 1.3568`
- `1652` training steps in `600122ms`
- `Total submission size int8+zlib: 15795501 bytes`
- conclusion:
- this was the first near-tie in tranche H
- replacing learned skip weights with unit skips only lost `0.0004`
- immediate implication:
- the skip topology itself may matter more than learning the skip weights
- `H4` is now the critical next run, because it separates “skip topology matters” from “skip weights matter”

- `AL-20260330-009` tested `H4`: `SKIP_MODE=off`

- `1659` training steps in `600100ms`
- `Total submission size int8+zlib: 15811518 bytes`
- conclusion:
- removing skip topology regressed clearly
- much more than the near-tie from `SKIP_MODE=unit`
- immediate implication:
- the skip path itself is useful
- learned skip weighting is the softer target than skip topology
- `H5` is still worth the compute because it is the only remaining interaction test for the residual-simplification package

- `AL-20260330-010` tested `H5`: `USE_RESID_MIX=0, USE_ATTN_SCALE=0, USE_MLP_SCALE=0, SKIP_MODE=unit`

## Tranche H Synthesis

- Tranche H did not find a new frontier run.
- But it did answer the residual question cleanly.
- `resid_mix` is real.
- learned residual scales are real.
- skip topology is real.
- learned skip weights may not be very important.

- The key pattern is:
- most residual simplifications were not near misses
- they were real regressions
- except `SKIP_MODE=unit`, which nearly tied the frontier

- That means the residual family is mostly closed as a generic simplification path.
- The one specific lesson worth carrying forward is:
- if we later want to simplify the block, we should target skip-weight parameterization before touching skip topology itself.

- So the next tranche should not be “more residual pruning.”
- The next tranche should be a bolder architecture change:
- latent-KV / MLA-lite attention compression
- or hybrid delta-style / state-space mixer layers

## Tranche I Start

- I implemented the first bold attention-architecture branch: latent-KV compression inside the attention block.
- The tranche is [`T-20260330-I`](../../agent_lab/tranches.md).
- Main question:
- can the model keep useful attention behavior if K/V structure is compressed through a latent bottleneck?

- Planned runs:
- `I1`: all-layer latent-KV with `LATENT_KV_DIM=128`
- `I2`: all-layer latent-KV with `LATENT_KV_DIM=64`
- `I3`: upper-layer latent-KV with `LATENT_KV_DIM=64`
- `I4`: lower-layer latent-KV with `LATENT_KV_DIM=64`
- `I5`: all-layer latent-KV `64` plus `10` layers

## Tranche I Progress

- `AL-20260330-011` tested `I1`: all-layer `LATENT_KV_DIM=128`
- `final_int8_ttt_lora val_bpb: 1.3718`
- `1647` training steps in `600216ms`
- `Total submission size int8+zlib: 15825767 bytes`
- `final_int8_zlib_roundtrip_exact val_bpb: 1.40750844`
- `TTT eval time: 801141ms`
- conclusion:
- the latent-KV idea is technically viable in this codebase
- it trained cleanly, stayed challenge-valid on size, and finished the full TTT path
- but it is not an immediate drop-in win
- the quality loss is too large against the `1.3564` anchor
- and it did not buy a cleaner eval path either

- immediate implication:
- do not conclude that latent-KV is dead yet
- but the next useful question is no longer “does mild all-layer latent-KV work?”
- it is “is the problem full-stack compression, or the latent-KV idea itself?”

- `AL-20260330-012` tested `I2`: all-layer `LATENT_KV_DIM=64`
- `final_int8_ttt_lora val_bpb: 1.3865`
- `1684` training steps in `600114ms`
- `Total submission size int8+zlib: 14726947 bytes`
- `final_int8_zlib_roundtrip_exact val_bpb: 1.42228367`
- `TTT eval time: 794736ms`
- conclusion:
- stronger all-layer latent-KV made the quality trade worse
- it did buy more headroom and slightly more steps than `I1`
- but the regression versus `I1` means “just compress harder everywhere” is not the right path

- immediate implication:
- tranche I is now much clearer scientifically
- the live hypothesis is no longer about latent width alone
- it is about placement
- `I3` and `I4` are now the important experiments, because they test whether full-stack compression is the real mistake
- I also updated `I5` to be the more meaningful reinvestment test:
- instead of `10L + latent_kv128`, it will now test `10L + latent_kv64`
- because `I2` is the branch that actually created substantial headroom

- `AL-20260330-013` tested `I3`: upper-only `LATENT_KV_DIM=64`
- `final_int8_ttt_lora val_bpb: 1.3685`
- `1672` training steps in `600081ms`
- `Total submission size int8+zlib: 15321329 bytes`
- `final_int8_zlib_roundtrip_exact val_bpb: 1.40297890`
- `TTT eval time: 804015ms`
- conclusion:
- this was the best latent-KV result in the tranche
- localizing compression to the upper layers recovered meaningful quality versus all-layer latent64
- the family is therefore placement-sensitive, not simply dead

- `AL-20260330-014` tested `I4`: lower-only `LATENT_KV_DIM=64`
- `final_int8_ttt_lora val_bpb: 1.3737`
- `1674` training steps in `600230ms`
- `Total submission size int8+zlib: 15300594 bytes`
- `final_int8_zlib_roundtrip_exact val_bpb: 1.40949261`
- `TTT eval time: 805020ms`
- conclusion:
- lower-only compression also beat the all-layer latent64 baseline
- but it trailed the upper-only placement
- so if latent-KV is used at all, the upper stack currently looks like the cleaner place to localize it

- `AL-20260330-015` tested `I5`: `10L` + all-layer `LATENT_KV_DIM=64`
- `final_int8_ttt_lora val_bpb: 1.3883`
- `1506` training steps in `600166ms`
- `Total submission size int8+zlib: 15840917 bytes`
- `final_int8_zlib_roundtrip_exact val_bpb: 1.42382513`
- `TTT eval time: 879933ms`
- conclusion:
- reinvesting the stronger all-layer latent-KV savings into one more layer did not rescue the line
- the extra layer gave back too many steps and made the evaluation path even slower

## Tranche I Synthesis

- Tranche I did not find a new frontier run.
- But it answered the latent-KV question well.
- The mechanism is not dead.
- The naive full-stack version is the wrong shape.
- Placement matters.

- The key pattern is:
- all-layer latent-KV `128` lost
- all-layer latent-KV `64` lost even more
- upper-only latent-KV `64` recovered the most quality
- lower-only latent-KV `64` also recovered some quality, but less
- extra depth on top of all-layer latent-KV `64` did not rescue the family

- So the real tranche-I conclusion is:
- full-stack K/V compression is too blunt for the current frontier
- if this family is worth revisiting, it should be revisited as a localized upper-stack design, not as a global compression pass
- but even the best localized version is still clearly behind the current best run

- immediate implication:
- latent-KV is now understood well enough to stop being the top priority
- the next bold tranche should probably pivot to a new mechanism family, with hybrid sequence mixers, output-head architecture, or local-global attention as the strongest next options
