## AttnRes-lite Implementation

I implemented code support for the previously planned `T-20260331-Q` tranche.

New trainer surface:

- `ATTNRES_LAYERS`
- `ATTNRES_NUM_SOURCES`
- `ATTNRES_ROUTING_MODE=token_softmax|shared_scalar`

What the implementation does:

- adds a lightweight `AttnResidualLite` router inside selected blocks
- on selected later layers, replaces fixed skip-style reuse with a softmax mix over a few earlier depth states
- uses the current stream plus a small set of earlier sources rather than full all-layer depth attention

Candidate source order in the current implementation:

- current stream
- matched encoder skip for that decoder layer
- shallow encoder anchor
- root stream `x0` if more sources are requested

This keeps the mechanism lightweight and aligned with the tranche hypothesis: test dynamic depth routing without turning it into a full expensive all-layer attention-over-depth system.

Validation:

- `python3 -m py_compile agent_lab/train_gpt.py`
- a direct forward-pass smoke test with AttnRes-lite enabled on late layers succeeded
- `python3 scripts/agent_lab/run_tranche.py agent_lab/tranche_manifests/20260331-Q-attnres-lite.json` dry-ran successfully


## Tranche P Closeout

Tranche `P` answered the right question. The original hybrid winner was good, but it was not “done.”

The five runs told a clean story:

- [`AL-20260331-001`](../../agent_lab/experiments.tsv) lowered the mixer footprint from four lower layers to three and still improved the frontier to `1.3453`. That means the hybrid idea is robust; it was not just one lucky lower-four setting.
- [`AL-20260331-002`](../../agent_lab/experiments.tsv) pushed the idea harder with five lower mixers. That run got smaller and faster, but lost enough quality to fall behind the best lower-four line. So the answer is not “replace as much lower-stack attention as possible.”
- [`AL-20260331-003`](../../agent_lab/experiments.tsv) shrank the mixer width and stayed exactly flat to the older winner. That is useful because it says the winning placement is not terribly fragile to a modest mixer shrink, but it did not improve the frontier.
- [`AL-20260331-004`](../../agent_lab/experiments.tsv) widened the mixer and became the new best valid run at `1.3451`.
- [`AL-20260331-005`](../../agent_lab/experiments.tsv) widened the mixer kernel and also beat the older winner, though not the wider-mixer result.

Simple reading:

- lower-stack mixers are definitely real
- the family still has headroom
- the best current story is not “more mixer layers,” it is “a stronger mixer inside the same lower-four placement”

That matters. It means the frontier now believes the lower stack can stay mixer-heavy, but the mixer itself still needed more strength.

## Tranche Q Closeout

Tranche `Q` gave a strong answer, and it was mostly negative.

The core question was whether fixed skip and residual routing should give way to a small input-dependent router over earlier layer states. On this frontier, the answer is: not in the broad late-layer form we tested.

- [`AL-20260331-007`](../../agent_lab/experiments.tsv) tried the richer late-layer version with four source states and collapsed badly to `1.5043`.
- [`AL-20260331-009`](../../agent_lab/experiments.tsv) tested the cheaper shared-routing version and was just as bad at `1.5133`.
- [`AL-20260331-010`](../../agent_lab/experiments.tsv) stacked AttnRes-lite on top of the best cheap-routing package and still failed badly at `1.5016`.

So the key reading is that the failure is not a small implementation tax. Broad late-layer dynamic depth routing is simply the wrong routing shape for the current hybrid frontier.

There was one interesting exception:

- [`AL-20260331-008`](../../agent_lab/experiments.tsv) only applied AttnRes-lite to the top two layers and came in at `1.3499`.

That is still worse than the current best [`AL-20260331-004`](../../agent_lab/experiments.tsv) at `1.3451`, so it is not a win. But it is much cleaner than the other Q-family results. That suggests a more limited interpretation:

- if dynamic depth routing is useful at all here, it likely belongs only at the very top of the stack
- it does not look like a good broad replacement for the current fixed routing scheme

So tranche `Q` mostly closes the first AttnRes-lite branch. The family is not fully dead, but it is no longer a priority frontier path in its current form. The only version worth remembering is a much lighter top-of-stack-only revisit later, if stronger families flatten.

## Tranche S: Hybrid Mixer Plus Cheap Routing Combo

This was the most justified immediate tranche, because we already had two independent positive signals:

- hybrid lower-stack mixers were the best architecture family
- cheap routing was the strongest secondary routing family

So the real scientific question was not whether either one worked. It was whether they were complementary or merely different ways of buying the same gain.

The answer was clear: they are complementary, but not equally so.

- [`AL-20260331-016`](../../agent_lab/experiments.tsv) showed that the full cheap-routing package stacks cleanly on the stronger hybrid winner and improves the frontier to `1.3434`.
- [`AL-20260331-017`](../../agent_lab/experiments.tsv) then improved further to `1.3429` using only shared scalar skip gates.
- [`AL-20260331-018`](../../agent_lab/experiments.tsv) also improved with only scalar `resid_mix`, which tells us the stronger hybrid backbone is broadly compatible with simplified routing.
- [`AL-20260331-019`](../../agent_lab/experiments.tsv) lost with shared residual scales alone, so that part is not the main mechanism.
- [`AL-20260331-020`](../../agent_lab/experiments.tsv) kept the combo strong while adding head-focused cooldown, but did not beat the simpler skip-gate line.

The simple reading is:

- the hybrid winner did want routing cleanup
- the single most important routing change was shared scalar skip gates
- the full routing package was good, but not better than the cleaner skip-gate-only version

So tranche `S` did not merely say "combos can help." It said something more precise:

- the model wants skip topology
- but it does not need a full learned skip-weight vector at every skip edge
- a much cheaper coarse skip gate is enough, and apparently better

That is strong architectural evidence, not generic optimization noise.

## Tranche T: Local-Global Attention Recovery

Before this tranche could be judged, the old local-attention crash had to be debugged. The problem turned out not to be conceptual at all. The trainer had flash SDP enabled and math SDP disabled, which is fine for the normal fast path but not for masked local attention. Once local attention was active, the compiled path had no valid backend and died immediately.

I fixed that by enabling math SDP as a fallback when local attention is active. After that, the family ran cleanly.

Scientifically, though, the results were mostly negative:

- [`AL-20260331-022`](../../agent_lab/experiments.tsv), local attention across the four upper attention layers, failed badly at `1.3813`.
- [`AL-20260331-023`](../../agent_lab/experiments.tsv), top-two local attention with a wider `512` window, was the best repaired run at `1.3545`, but still nowhere near the frontier.
- [`AL-20260331-024`](../../agent_lab/experiments.tsv), alternating local attention through the remaining attention stack, also lost clearly.
- [`AL-20260331-025`](../../agent_lab/experiments.tsv), top-two local attention with a narrower `128` window, also lost.

So this tranche did answer the question. The answer was:

- on this hybrid winner, once the lower stack is already cheaper and more structured, the remaining attention layers still seem to want global access

There is one nuance. The best repaired run was the top-two `512`-window variant, which says locality is not purely nonsensical. But it still lost too much to remain a priority branch.

So the real conclusion is not "local attention can never work." It is:

- plain local-window replacement of the remaining attention layers is not the next frontier move on this backbone

## Tranche U: Compression-Native Backbone Audit

This was one of the most important tranches conceptually, because it attacked a known weakness in our program. The public frontier strongly suggests that compression is not just a final serialization step. It is part of the architecture story.

The first mechanism I tested here was low-rank factorization, both in attention and in the MLP.

The result was very clear:

- factorizing attention at rank `128` or `64` saved bytes, but lost too much quality to matter
- factorizing the MLP was disastrous
- combining attention and MLP factorization did not rescue anything

Concretely:

- [`AL-20260331-026`](../../agent_lab/experiments.tsv) and [`AL-20260331-027`](../../agent_lab/experiments.tsv) were clear attention-factorization losses
- [`AL-20260331-028`](../../agent_lab/experiments.tsv) and [`AL-20260331-029`](../../agent_lab/experiments.tsv) showed that the MLP is far more sensitive to naive factorization than the attention block is
- [`AL-20260331-030`](../../agent_lab/experiments.tsv) confirmed that combining the two only produces a somewhat smaller but still clearly weaker model

So this tranche did not falsify compression-aware architecture as a direction. It falsified one specific naive story:

- straightforward low-rank factorization is not enough

That is still useful. It tells us the next compression-native branch must be structurally different. It should not just be "pick another rank."

## Tranche V: Top-Only Dynamic Depth Routing Revisit

This was the disciplined revisit of AttnRes-lite.

The earlier tranche `Q` already told us that broad late-layer dynamic routing was the wrong shape. The only weak positive signal had come from applying the mechanism only at the top of the stack. So this tranche asked a much narrower question:

- if dynamic depth routing is useful at all, can it work when kept tiny, top-only, and source-limited?

This time the answer was yes.

- [`AL-20260331-031`](../../agent_lab/experiments.tsv), a final-layer two-source router, improved to `1.3440`
- [`AL-20260331-032`](../../agent_lab/experiments.tsv), top-two layers with two sources, also improved to `1.3443`
- [`AL-20260331-035`](../../agent_lab/experiments.tsv), top-two routing plus shared skip gates, reached `1.3433`

The negative controls were just as important:

- [`AL-20260331-033`](../../agent_lab/experiments.tsv) showed that top-only shared-scalar routing loses the gain
- [`AL-20260331-034`](../../agent_lab/experiments.tsv) showed that adding a third source to the final-layer router also hurts

So the revived AttnRes-lite family now has a clear shape:

- top-only
- token-wise
- very few source states

This is one of the most useful scientific outcomes in the whole program. The family went from "mostly falsified" to "real, but only under very tight placement and complexity constraints."

It still did not beat the new skip-gate winner, so it is not the main frontier. But it is now a live secondary family again.

## Tranche W: Broad MLP Family Audit

This tranche asked a very simple but important question:

- is the current `relu^2` MLP actually special, or have we been carrying it forward without properly testing broad alternatives?

The answer was much cleaner than I expected:

- plain `ReLU` lost
- `SiLU` lost
- `GELU` lost
- the surfaced `relu^2` path stayed strong

That means the current frontier really does want the squared-ReLU family. This was not just inherited baggage.

There was one interesting exception:

- [`AL-20260331-040`](../../agent_lab/experiments.tsv), a gated SiLU MLP, was competitive on `val_bpb`

But it blew through the size cap. That means it is not a live immediate frontier result, but it is still a useful research clue:

- gating may be interesting
- broad smooth activations are not

So the right lesson from this tranche is not "never revisit the MLP." It is:

- if the MLP is revisited, do not waste time on plain smooth-family swaps
- focus on gated capacity under tighter size control

## Tranche X: Polynomial MLP Audit

This tranche went deeper into the MLP question, but only inside the polynomial family.

Here the pattern was also clear:

- cubic-heavy forms were bad
- the only live non-anchor variant was mixing linear and quadratic behavior

Concretely:

- [`AL-20260331-042`](../../agent_lab/experiments.tsv), pure `relu^3`, regressed badly
- [`AL-20260331-044`](../../agent_lab/experiments.tsv), `relu^2` plus a cubic tail, also regressed
- [`AL-20260331-045`](../../agent_lab/experiments.tsv), norm-before-square, stayed close but did not improve
- [`AL-20260331-043`](../../agent_lab/experiments.tsv), `relu + 0.5 * relu^2`, was the best non-anchor result at `1.3444`

So the polynomial family now has a very simple lesson:

- if there is more headroom in the MLP, it is on the "linear plus quadratic" side
- not on the cubic side

That gives us a much narrower and more useful next MLP branch than we had before.

## Overall Scientific Synthesis

This full program taught a coherent architectural story.

The frontier now looks like:

- lower stack simplified by sequence mixers
- upper reasoning stack preserved
- routing cleaned up, but only in the simplest useful way
- dynamic depth routing allowed only at the very top and only in a very narrow form
- the MLP family still anchored on `relu^2`
- compression-aware architecture still unresolved, with naive low-rank factorization mostly ruled out

In other words, the model keeps rewarding the same principle:

- **use structure and simplicity where the model can afford it**
- **but do not spread extra cleverness across the whole stack**

That principle showed up repeatedly:

- lower-stack mixers worked, but broader replacement did not
- skip gates worked, but more complicated routing pieces were weaker
- top-only tiny depth routing worked, but broad late-layer routing collapsed
- `relu^2` worked, but broad smooth-family swaps and cubic-heavy forms did not

So the frontier theory is getting sharper:

- the model likes specialization by depth
- it likes lighter mechanisms when they are carefully placed
- it dislikes broad complexity spread across too much of the network

The next serious moves should come from that theory, not from generic tuning.

## Code Support For Y / Z / AB / AA

I then implemented the missing trainer support in [`train_gpt.py`](../../agent_lab/train_gpt.py) so those four tranches are actually runnable.

What changed, simply:

- `Y` support: new `MLP_STRUCTURE` variants let the model run no-expand quadratic token MLPs, no-expand linear-plus-quadratic token MLPs, and depth-selective lower-light / top-full MLP plans.

So the sharing tranche measures a real compressed-artifact effect, not a fake “shared at runtime but duplicated on disk” trick.

## Z1 Crash Investigation

When the live `YZABAA` program reached [`AL-20260401-051`](../../agent_lab/experiments.tsv), the failure turned out not to be in `AA` at all. It was the first `Z` run: the lower-two `mixer_only` block recipe.

The exact failure was a DDP unused-parameter error during warmup:

- `RuntimeError: Expected to have finished reduction in the prior iteration before starting a new one`
- unused parameter indices corresponded to the dead MLP-scale parameters on mixer-only blocks

The mechanism bug was simple:

- I removed the FFN path for `mixer_only` blocks
- but I still instantiated `mlp_scale` on those blocks
- so DDP saw trainable parameters that never participated in the loss

The fix was correspondingly narrow in [`train_gpt.py`](../../agent_lab/train_gpt.py):

- only create `mlp_scale` when `use_mlp_scale` is enabled **and** the block actually has an MLP

That preserves the block-uniformity experiment while removing the fake dead parameter path. The live `AB` process was already running on the old loaded code, so the fix applies to subsequent reruns of `Z` and anything after it, not retroactively to the already-started worker.
